# Phase 7: SEO Foundation & Build Infrastructure - Research

**Researched:** 2026-02-04
**Domain:** SEO for React SPAs, Build Optimization, Structured Data
**Confidence:** MEDIUM-HIGH

## Summary

Phase 7 adds SEO foundation and build infrastructure to a React 18 + TypeScript + Vite SPA deployed to GitHub Pages. The research focused on three domains: (1) SEO meta tags and structured data implementation for React SPAs, (2) build-time optimizations with Brotli compression, and (3) favicon/icon requirements for modern web and PWA.

The standard approach for 2026 combines static meta tags in index.html with JSON-LD structured data for rich search results, vite-plugin-sitemap for automatic sitemap generation, and vite-plugin-compression for Brotli pre-compression. Since GitHub Pages is static hosting without SSR, the site will use static meta tags (which work for bots) plus client-side updates with react-helmet-async for dynamic content.

**Key constraint:** GitHub Pages does not support server-side rendering. This means social media crawlers and search bots will see the initial HTML meta tags from index.html, not dynamically updated tags from React components. This is acceptable for a single-page portfolio site where the meta tags remain constant across all sections.

**Primary recommendation:** Implement static meta tags in index.html for SEO/social preview, add JSON-LD Organization schema via script tag, generate sitemap.xml and robots.txt at build time with vite-plugin-sitemap, enable Brotli compression with vite-plugin-compression, and provide minimal modern favicon set (favicon.ico, icon.svg, apple-touch-icon.png) plus web manifest for PWA capability.

### Pattern 1: Static Meta Tags in index.html

**What:** Place all SEO meta tags directly in index.html's `<head>` section
**When to use:** GitHub Pages deployment without SSR
**Why:** Social media crawlers (Facebook, Twitter) and search bots execute minimal JavaScript and primarily parse the initial HTML response. On GitHub Pages, the initial HTML is all they see.

**Example:**

```html
<!-- index.html -->
<head>
  <meta charset="UTF-8" />
  <title>Puff Puff Dev | Software Development</title>
  <meta name="description" content="Turning your ideas into polished software that users love." />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />

  <!-- Canonical URL -->
  <link rel="canonical" href="https://puffpuff.dev/" />

  <!-- Open Graph -->
  <meta property="og:type" content="website" />
  <meta property="og:url" content="https://puffpuff.dev/" />
  <meta property="og:title" content="Puff Puff Dev | Software Development" />
  <meta property="og:description" content="Turning your ideas into polished software that users love." />
  <meta property="og:image" content="https://puffpuff.dev/og-image.png" />
  <meta property="og:site_name" content="Puff Puff Dev" />

  <!-- Twitter Card -->
  <meta name="twitter:card" content="summary_large_image" />
  <meta name="twitter:title" content="Puff Puff Dev | Software Development" />
  <meta name="twitter:description" content="Turning your ideas into polished software that users love." />
  <meta name="twitter:image" content="https://puffpuff.dev/og-image.png" />

  <!-- Favicons -->
  <link rel="icon" href="/favicon.ico" sizes="32x32" />
  <link rel="icon" href="/icon.svg" type="image/svg+xml" />
  <link rel="apple-touch-icon" href="/apple-touch-icon.png" />
  <link rel="manifest" href="/manifest.webmanifest" />
</head>
```

### Pattern 2: JSON-LD Structured Data in React Component

**What:** Render JSON-LD structured data as a script tag using React's dangerouslySetInnerHTML
**When to use:** Adding Organization, Person, or other schema.org types for rich search results
**Why:** Google explicitly prefers JSON-LD over Microdata/RDFa. React requires dangerouslySetInnerHTML because it escapes quotes by default.

**Example:**

```tsx
// src/components/StructuredData.tsx
import React from 'react';

export const OrganizationSchema: React.FC = () => {
  const schema = {
    "@context": "https://schema.org",
    "@type": "Organization",
    "name": "Puff Puff Dev",
    "url": "https://puffpuff.dev/",
    "logo": "https://puffpuff.dev/icon-512.png",
    "sameAs": [
      "https://github.com/romatroskin"
    ],
    "description": "Turning your ideas into polished software that users love."
  };

  return (
    <script
      type="application/ld+json"
      dangerouslySetInnerHTML={{ __html: JSON.stringify(schema) }}
    />
  );
};
```

**Usage in App:**

```tsx
// src/App.tsx
import { OrganizationSchema } from './components/StructuredData';

function App() {
  return (
    <>
      <OrganizationSchema />
      {/* Rest of app */}
    </>
  );
}
```

### Pattern 3: Vite Plugin Configuration for Sitemap + Compression

**What:** Configure Vite plugins in vite.config.ts to generate sitemap.xml, robots.txt, and pre-compress assets
**When to use:** Production builds for static hosting

**Example:**

```typescript
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import Sitemap from 'vite-plugin-sitemap';
import viteCompression from 'vite-plugin-compression';

export default defineConfig({
  plugins: [
    react(),

    // Generate sitemap.xml and robots.txt
    Sitemap({
      hostname: 'https://puffpuff.dev',
      dynamicRoutes: ['/'], // Single-page app, but can add sections
      changefreq: 'monthly',
      priority: 1.0,
      lastmod: new Date().toISOString().split('T')[0],
      generateRobotsTxt: true,
      robots: [
        {
          userAgent: '*',
          allow: '/',
        }
      ]
    }),

    // Brotli compression (primary)
    viteCompression({
      algorithm: 'brotliCompress',
      ext: '.br',
      threshold: 1024, // Only compress files > 1KB
      deleteOriginFile: false,
      compressionOptions: {
        level: 11 // Maximum compression for static assets
      }
    }),

    // Gzip compression (fallback)
    viteCompression({
      algorithm: 'gzip',
      ext: '.gz',
      threshold: 1024,
      deleteOriginFile: false
    })
  ]
});
```

### Anti-Patterns to Avoid

- **Using react-helmet-async without static fallback on GitHub Pages:** Crawlers won't execute React, so dynamic meta tags are invisible. Always include static tags in index.html as the source of truth.
- **Blocking JavaScript/CSS in robots.txt for SPAs:** Search engines need to execute JavaScript to render SPAs. Never disallow .js or .css files.
- **Hand-rolling sitemap.xml without lastmod dates:** Search engines use lastmod to prioritize crawling. Use a plugin that auto-generates with current dates.
- **Multiple conflicting canonical tags:** Only one canonical URL per page. Don't add canonical via both index.html and react-helmet-async.
- **Using fragment identifiers (#/) for routes:** Google can handle them but they're less reliable. Use proper HTML5 history routing (React Router in browser history mode).

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sitemap generation | Custom Node script to write XML | vite-plugin-sitemap | Handles URL escaping, date formatting, robots.txt generation, lastmod tracking |
| Asset compression | Manual gzip/Brotli build step | vite-plugin-compression | Integrates with Vite build, handles thresholds, preserves originals, supports multiple algorithms |
| Favicon generation | Manual PNG exports at each size | Favicon generator service (favicon.io, RealFaviconGenerator) | Generates all required sizes, formats, and platform-specific variants |
| JSON-LD schema | String concatenation | Type-safe schema builder or schema.org examples | Ensures valid JSON, handles escaping, validates against schema.org types |
| OG image creation | CSS-to-image at runtime | Static design tool (Figma, Canva) → PNG export | Runtime generation adds complexity; static images are cacheable and reliable |

**Key insight:** SEO infrastructure has mature tooling. The complexity is in edge cases (URL encoding, MIME types, compression levels, image optimization). Use established plugins that handle these correctly.

## Common Pitfalls

### Pitfall 1: GitHub Pages Doesn't Execute React for Crawlers

**What goes wrong:** Developers assume react-helmet-async will update meta tags for social media crawlers, but Facebook/Twitter/LinkedIn bots see only the initial HTML from index.html, not client-side React updates.

**Why it happens:** GitHub Pages serves static HTML without server-side rendering. Crawlers parse the initial response and don't wait for JavaScript execution. They see the empty `<div id="root"></div>` and meta tags from index.html only.

**How to avoid:**
1. Put all critical SEO meta tags (title, description, OG tags, Twitter cards) in index.html as the source of truth
2. Use react-helmet-async only for enhancing UX (page title updates for users) or if you add SSR/pre-rendering later
3. Test with Facebook Sharing Debugger and Twitter Card Validator to verify they see correct tags

**Warning signs:**
- Social media previews show generic GitHub Pages title instead of your site title
- Facebook Debugger shows "Missing og:image" despite having it in React Helmet
- Twitter Card Validator shows fallback text instead of your description

**Source:** [GitHub Pages SEO with react-helmet-async](https://www.creolestudios.com/how-to-make-react-website-seo-friendly/)

### Pitfall 2: Robots.txt Blocking Critical Assets

**What goes wrong:** Developers add `Disallow: /*.js` or `Disallow: /*.css` to robots.txt, preventing search engines from rendering the SPA correctly, leading to empty search results.

**Why it happens:** Misunderstanding that SPAs require JavaScript execution for content. Traditional SEO advice to block assets doesn't apply to client-side rendered apps.

**How to avoid:**
1. In robots.txt, only disallow truly sensitive paths (admin panels, API routes if exposed)
2. Always allow JavaScript and CSS: `Allow: /*.js` and `Allow: /*.css`
3. Test with Google Search Console's "Fetch as Google" tool to verify your main bundle loads
4. Use sitemap.xml to explicitly guide crawlers to important sections

**Warning signs:**
- Google Search Console reports "Page is not indexable" or "Redirect error"
- Lighthouse SEO audit fails with "Document doesn't have a valid robots.txt"
- Site appears in search but with no description or shows "Loading..."

**Source:** [Setting up React app for Google's SEO crawlers](https://vonkunesnewton.medium.com/setting-up-your-react-app-for-googles-seo-crawlers-d16b102b0103)

### Pitfall 3: Wrong Open Graph Image Dimensions

**What goes wrong:** OG images appear cropped, pixelated, or rejected by social platforms due to incorrect dimensions or file size.

**Why it happens:** Different platforms have different requirements. Facebook requires minimum 1200x630px, Twitter prefers 2:1 ratio, and LinkedIn has file size limits.

**How to avoid:**
1. Use exactly 1200x630px (1.91:1 aspect ratio) for cross-platform compatibility
2. Keep file size under 8MB (ideally < 300KB for fast loading)
3. Use PNG or JPG format (PNG for graphics with text, JPG for photos)
4. Test the image by posting the URL on each platform (Facebook, Twitter, LinkedIn)
5. Ensure image includes visible text/branding since thumbnails can be small

**Warning signs:**
- Image appears zoomed-in or cropped oddly when shared
- Social platform shows generic placeholder instead of your image
- Image takes multiple seconds to load in preview
- Text in image is unreadable at thumbnail size

**Sources:**
- [Open Graph Image Dimensions Guide](https://www.oreateai.com/blog/understanding-open-graph-image-dimensions-a-guide-for-social-media-success/)
- [Open Graph Image Size Best Practices](https://doinwp.com/og-image-size-and-requirements/)

### Pitfall 4: Missing apple-touch-icon Causes iOS Screenshot Fallback

**What goes wrong:** iOS users who "Add to Home Screen" see a miniature screenshot of the page instead of a proper icon, looking unprofessional.

**Why it happens:** iOS requires a specific 180x180px PNG file named apple-touch-icon.png. If missing, iOS falls back to taking a screenshot.

**How to avoid:**
1. Create a 180x180px PNG icon specifically for iOS (can be same design as other icons)
2. Place in public/apple-touch-icon.png (iOS auto-detects this path) OR explicitly link in HTML: `<link rel="apple-touch-icon" href="/apple-touch-icon.png">`
3. Use PNG format only (SVG, ICO, and other formats are not supported by iOS)
4. Ensure icon has proper padding (iOS adds rounded corners and shadow)
5. Test on actual iOS device or simulator

**Warning signs:**
- Home screen icon shows zoomed-out webpage screenshot
- Icon looks pixelated or too small
- Different icon appears on Android vs iOS

**Source:** [Apple Touch Icons Guide 2026](https://favicon-generator.ai/info/apple-touch-icons)

### Pitfall 5: Brotli Assets Not Served Due to Missing MIME Configuration

**What goes wrong:** Server generates .br files but browsers don't receive them, missing out on 15-25% additional compression vs gzip.

**Why it happens:** Static hosting (including GitHub Pages) may not auto-detect .br files or set correct `Content-Encoding: br` headers.

**How to avoid:**
1. Verify your hosting supports pre-compressed Brotli (GitHub Pages does support it)
2. Keep both .br and .gz versions for maximum compatibility
3. Configure server to serve .br files with `Content-Encoding: br` header (GitHub Pages handles this automatically)
4. Test in browser DevTools Network tab: check for `.br` extension and `Content-Encoding: br` response header
5. Fallback to gzip if Brotli isn't supported by hosting

**Warning signs:**
- Build generates .br files but browser downloads .js instead of .js.br
- Network tab shows large transfer sizes despite having compressed files
- `Content-Encoding` header is missing or shows `identity`

**Source:** [Vite.js Brotli pre-compression](https://bytepursuits.com/vite-js-to-precompress-your-js-and-css-bundles-in-brotli-and-gzip-format)

**Note on GitHub Pages:** GitHub Pages CDN (Fastly) automatically serves pre-compressed files (.br and .gz) when present and client accepts them via `Accept-Encoding` header. No additional configuration needed. However, the compression must happen at build time since GitHub Pages can't compress on-the-fly.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Dozens of favicon sizes | Minimal set: favicon.ico, icon.svg, apple-touch-icon.png, 2 PWA icons | ~2021-2022 | Reduces complexity, modern browsers scale intelligently |
| react-helmet | react-helmet-async | 2020 | Supports React 18 concurrent rendering, better SSR, no memory leaks |
| Manual robots.txt + sitemap.xml | vite-plugin-sitemap auto-generation | 2022-2023 | Build-time generation ensures lastmod dates are current |
| Runtime og:image generation | Static pre-designed og:image | Ongoing | Static is faster, more reliable, cacheable; runtime adds complexity |
| Microdata/RDFa for schema | JSON-LD only | 2015-2018 | Google prefers JSON-LD, easier to implement and validate |
| Server-side Brotli | Build-time pre-compression | 2020+ | Required for static hosting, works on GitHub Pages/Netlify/Vercel |

**Deprecated/outdated:**
- **react-helmet (non-async):** Causes memory leaks with SSR and doesn't support React 18 concurrent features. Use react-helmet-async instead.
- **Manual favicon.png in 16, 32, 48, 64, 96, 128, 256 sizes:** Browsers now scale intelligently from 32x32 ICO and SVG. Only need specific sizes for PWA manifest (192, 512) and iOS (180).
- **PNG-only favicons:** SVG favicons are now standard for modern browsers, with ICO fallback for legacy.
- **Separate light/dark favicon files:** Single SVG with `prefers-color-scheme` media query is cleaner than separate files + JavaScript detection.

## Open Questions

Things that couldn't be fully resolved:

1. **Vite Plugin Compression: vite-plugin-compression vs vite-plugin-compression2**
   - What we know: vite-plugin-compression is most popular (3.5k stars), last updated 2 years ago. vite-plugin-compression2 is newer fork with better maintenance.
   - What's unclear: Which is more reliable for production builds in 2026? Does compression2 have breaking changes?
   - Recommendation: Start with vite-plugin-compression (more battle-tested). Monitor build output for errors. If issues arise, switch to compression2.
   - **Confidence:** MEDIUM - Both should work, but older plugin has more production usage.

2. **GitHub Pages Brotli Serving Verification**
   - What we know: GitHub Pages (via Fastly CDN) claims to serve pre-compressed files automatically.
   - What's unclear: Is there any configuration needed? Do custom domains work the same?
   - Recommendation: After deploying, verify with Chrome DevTools Network tab that .br files are served with `Content-Encoding: br` header. If not detected, verify .br files exist in dist/ and check Accept-Encoding request header.
   - **Confidence:** MEDIUM-HIGH - Documentation confirms support, but real-world testing needed.

3. **Maskable Icon Requirement for PWA**
   - What we know: Maskable icons (with extra padding) adapt to different Android launcher shapes. PWA specs mention them.
   - What's unclear: Is maskable icon required for Phase 7 success criteria, or is it optional enhancement?
   - Recommendation: Include maskable icon in web manifest (`purpose: "maskable"`) for completeness. Generate with 20% safe zone padding.
   - **Confidence:** MEDIUM - Best practice for PWA, but not strictly required for basic functionality.

4. **Pre-rendering for GitHub Pages SEO**
   - What we know: prerender-spa-plugin and react-snap can generate static HTML for each route, improving SEO for multi-page SPAs.
   - What's unclear: Is pre-rendering necessary for a single-page portfolio where all content loads immediately? What's the ROI vs complexity?
   - Recommendation: Skip pre-rendering for Phase 7. The site is a single page with all content visible after initial load. Static meta tags in index.html are sufficient. Revisit if adding multi-page blog or docs sections.
   - **Confidence:** HIGH - Pre-rendering adds complexity without clear benefit for single-page portfolio.
